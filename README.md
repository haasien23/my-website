# 基于集群的多任务协同态势感知平台

本项目是一个面向本科毕业设计的无人机集群协同感知仿真平台，题目为“基于集群的多任务协同态势感知平台”。平台围绕多无人机、多目标、障碍环境、通信受限、节点故障、信息融合和算法对比等场景展开，通过 Web 页面把抽象的协同态势感知过程可视化、可配置化和可复现实验化。

简单来说，它不是只输出一组计算结果的脚本，而是一个完整的仿真展示系统：用户可以在浏览器中配置任务规模、感知半径、通信半径、天气扰动、障碍物数量、故障率、通信丢包率等参数，然后逐步运行仿真，观察 RADS 动态子群、随机派遣、全量派遣三种策略在同一复杂场景下的感知效果、派机过程、定位误差、能耗和任务成功率。

## 项目定位

这个毕设的核心问题是：在复杂环境中，如何让无人机集群根据任务需求动态选择合适的无人机子群，而不是让所有无人机盲目参与或随机派遣，从而在态势感知精度、资源消耗、任务响应和系统鲁棒性之间取得更好的平衡。

平台把这个问题拆成三个层面：

1. 场景层：构建多无人机、多目标、障碍物、天气、通信延迟、丢包、故障等约束共同存在的仿真环境。
2. 算法层：实现 RADS 动态子群调度策略，并提供随机派遣和全量派遣作为对照。
3. 展示层：用 Web 页面、3D 态势沙盘、指标卡、任务表格、趋势曲线和 CSV 导出支撑论文实验分析与答辩展示。

## 服务对象

本平台主要服务于以下几类对象：

- 毕设作者：用于演示毕业设计系统成果，支撑论文中的系统设计、算法设计、实验对比和结果分析。
- 答辩评委：通过页面直接观察算法运行过程，理解“为什么需要动态子群调度”以及“RADS 相比随机派遣和全量派遣的改进点”。
- 无人机集群与多智能体方向学习者：用于学习多目标任务分配、协同感知、传感器噪声、通信受限、障碍物绕行等基础建模方法。
- 后续开发者：可以在现有仿真框架上替换调度策略、增加新的融合算法、扩展指标体系或接入更真实的地图与任务模型。

## 核心功能

### 1. 多无人机多目标仿真

平台在二维任务区域中生成多架无人机和多个动态目标。每架无人机具有位置、高度、航向、速度偏置、感知能力偏置、通信能力偏置、剩余能量、故障状态和当前任务分配。每个目标具有位置、速度、优先级、不确定度和最近一次估计结果。

仿真不是静态场景。目标会在每个时间步持续移动，并在接近边界或障碍物时改变运动方向；无人机会根据当前分配任务向目标或绕障路径点移动；节点能量会随移动、感知、通信和空闲状态逐步下降。

### 2. 障碍物与绕障路径规划

平台将任务区域划分为网格地图，障碍物会占用部分网格并具有高度信息。障碍物不仅影响可视化效果，也会影响算法计算：

- 无人机到目标的可达距离不只使用欧氏距离，还会结合网格最短路近似绕障距离。
- 目标和无人机之间会进行视线遮挡判断，障碍物遮挡时感知会失败。
- 目标移动时会避开障碍区域，避免直接穿越不可通行网格。
- 无人机机动时会从距离图中选择下一个更接近目标的网格路径点。

路径规划相关逻辑位于 [`src/simulation/pathfinding.py`](src/simulation/pathfinding.py)，包括网格坐标转换、障碍判断、距离图构建、下一路径点选择和视线检测。

### 3. RADS 动态子群调度

RADS 是本项目中的主算法，含义为 Risk-Aware Dynamic Scheduling，可以理解为“风险感知的动态调度”。它的目标不是固定派出某些无人机，也不是每次派出全部无人机，而是根据当前任务紧急程度、目标位置、无人机能量、链路质量、路径距离和视线条件动态选择参与感知的无人机子群。

RADS 的调度过程大致如下：

1. 计算目标需求：每个目标根据优先级、不确定度、运动速度和随机扰动得到任务紧急度，再转换为需要的无人机数量。
2. 计算全局派遣预算：系统根据可用无人机数量、平均剩余能量、平均链路质量、热点目标数量和近期任务达标率，决定当前时间步最多派出多少架无人机。
3. 计算候选效用：对每个“无人机-目标”组合计算效用分数，分数综合考虑覆盖能力、剩余能量、通信链路、任务紧急度、是否进入有效感知范围、是否有无遮挡视线、旅行代价和冗余惩罚。
4. 贪心构建子群：优先处理紧急度更高的目标，在预算范围内为每个目标选择效用最高且尚未被选中的无人机。
5. 逐步重规划：每个仿真步都会重新计算任务需求和派遣结果，因此子群会随目标运动、节点故障、能量变化、链路变化和遮挡情况动态变化。

核心实现位于 [`src/simulation/core.py`](src/simulation/core.py) 中的 `assign_smart`、`compute_target_demand`、`compute_adaptive_budget` 和 `compute_utility` 等函数。

### 4. 对照策略

为了让实验结果更有说服力，平台同时实现了两种对照策略：

- 随机派遣：使用与 RADS 相同的任务需求和派遣预算，但在可行候选集合中随机选择无人机。它用于体现“只控制派机规模但不做智能选择”的效果。
- 全量派遣：尽可能让所有可用无人机参与任务分配。它用于观察高资源投入下的感知表现、能耗代价和信息时效。

三种策略可以同步运行，也可以在页面上选择单独运行某一种策略。同步运行时，三种策略使用相同的基础场景，便于做公平对比。

### 5. 协同感知与信息融合

无人机被派遣到目标后，并不一定能成功形成有效观测。一次观测需要同时满足多个条件：

- 目标处于无人机有效感知半径内。
- 目标没有被障碍物遮挡。
- 目标位于无人机传感器视场角内。
- 随机探测过程成功。
- 通信链路未因丢包而丢失观测结果。
- 若设置通信延迟，观测结果需要等待指定步数后才能参与融合。

成功到达融合阶段的观测会加入测量噪声。平台根据测量方差、链路质量和剩余能量为观测分配权重，再进行加权融合，得到目标位置估计值。融合前还会进行粗估计、残差抑制和聚类筛选，降低离群观测对最终结果的影响。

任务成功不是只看“有没有观测”，而是同时考虑确认数量、定位误差和一致性：

- 覆盖率：当前目标是否获得至少一次有效观测。
- 定位达标率：融合误差是否低于成功阈值。
- 确认率：有效观测数量是否达到任务所需确认数。
- 严格成功率：同时满足确认、误差达标和一致性要求。
- 平均融合误差：融合估计位置与真实目标位置之间的平均距离。
- 平均信息时效：参与融合的观测从采集到使用之间的平均延迟。
- 累计能耗：移动、感知、通信和空闲状态消耗的能量总和。

### 6. 天气、通信和故障扰动

平台支持不同天气预设，包括晴空、雾霾、降雨、风暴等。天气会影响传感器半径、视场、噪声、探测概率、链路质量、丢包率、通信延迟、无人机速度、移动能耗、故障概率和目标转向幅度。

通信方面，系统支持设置通信半径、丢包率和延迟步数。链路质量由邻居密度、中心位置、节点通信偏置和天气影响共同决定。故障方面，每架无人机会因为随机故障或能量过低变为不可用状态。

### 7. Web 可视化与交互

页面提供完整的实验交互能力：

- 左侧参数面板：配置无人机数量、目标数量、障碍物数量、感知半径、视场角、通信半径、丢包率、通信延迟、速度、噪声、融合半径、故障率、派机比例、运行步数、天气和随机种子。
- 顶部指标卡：实时展示 RADS、随机派遣、全量派遣的成功率、定位达标率、平均误差、累计能耗、平均派机数等指标。
- 3D 态势沙盘：基于 Three.js 展示地形、障碍物、无人机、目标、感知范围、任务连线和轨迹。
- 回放控制：仿真逐步推进，前端按时间步保存帧数据，用户可以观察不同阶段的任务变化。
- 并排对比窗口：可以同时观察三种策略在同一回放帧下的差异。
- 热点目标与任务明细：展示每个目标的优先级、不确定度、派机子群、观测数量、误差和任务状态。
- 自动结论文本：根据当前结果生成一段可用于论文分析的实验描述。
- CSV 导出：导出当前参数、策略指标、任务级结果和精细对比数据，便于整理实验表格。

## 系统架构

项目采用轻量级前后端结合架构：

```text
浏览器页面
  ├─ 参数输入、运行控制、3D 沙盘、指标图表、CSV 导出
  ↓ HTTP/JSON
Python Web 服务
  ├─ 路由处理
  ├─ 静态资源与模板渲染
  └─ 会话式仿真管理
  ↓
仿真内核
  ├─ 场景生成
  ├─ RADS / 随机派遣 / 全量派遣
  ├─ 路径规划与视线判定
  ├─ 目标运动与无人机移动
  ├─ 观测、通信、延迟和融合
  └─ 指标统计与帧数据序列化
```

后端使用 Python 标准库的 `ThreadingHTTPServer` 提供 HTTP 服务，使用 Jinja2 渲染页面模板。前端使用原生 JavaScript、Canvas 和本地打包的 Three.js 模块实现交互与 3D 展示，不依赖额外的前端构建步骤。

## 仿真运行流程

一次完整仿真由多个时间步组成，每个时间步主要执行以下流程：

1. 根据用户参数创建地图、障碍物、无人机和目标。
2. 根据当前目标位置构建每个目标对应的网格距离图。
3. 更新目标运动状态，处理边界和障碍物转向。
4. 根据运行模式分别执行 RADS、随机派遣或全量派遣。
5. 无人机根据分配任务向目标或绕障路径点移动。
6. 计算传感范围、视场、遮挡、探测概率、通信丢包和通信延迟。
7. 对有效观测进行加权融合，得到目标位置估计。
8. 更新任务成功率、覆盖率、定位误差、能耗、派机数量和信息时效。
9. 更新节点故障状态。
10. 记录历史指标和当前帧，返回给前端显示。

Web 端不是一次性等待所有步骤完成，而是通过会话接口逐步推进。这样页面进度条、曲线、3D 画面和任务表格都会持续刷新，更适合现场演示。

## 运行方式

### 1. 准备环境

建议使用 Python 3.10 或更新版本。后端主要依赖 Python 标准库，额外需要安装 Jinja2：

```powershell
pip install jinja2
```

前端 Three.js 文件已经放在 `static/vendor/` 下，不需要执行 `npm install`。

### 2. 启动服务

在项目根目录运行：

```powershell
python app.py
```

默认地址为：

```text
http://127.0.0.1:8000
```

也可以指定主机和端口：

```powershell
python app.py --host 127.0.0.1 --port 8000
```

### 3. 使用页面

打开浏览器访问默认地址后，可以按以下步骤演示：

1. 在左侧配置无人机数量、目标数量、障碍物数量、天气、故障率和运行步数。
2. 选择运行模式，推荐答辩时使用“三策略对比”。
3. 点击“运行仿真”，观察进度条、3D 沙盘和指标卡逐步更新。
4. 点击沙盘可以打开单独视图，使用拖拽、滚轮和双击观察 3D 场景。
5. 查看任务级精度对照表，比较不同策略对同一目标的派机数量、误差和状态。
6. 点击“导出实验 CSV”，保存当前实验结果用于论文表格或附录。

## API 说明

后端提供少量 JSON 接口，供前端按会话方式推进仿真：

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `GET` | `/` | 返回主页面 |
| `GET` | `/api/default-config` | 获取默认仿真参数 |
| `POST` | `/api/sessions` | 根据请求参数创建新的仿真会话 |
| `POST` | `/api/sessions/{session_id}/advance` | 将指定会话向前推进若干步 |
| `DELETE` | `/api/sessions/{session_id}` | 删除指定仿真会话 |

`POST /api/sessions/{session_id}/advance` 每次最多推进 5 步，前端当前按 1 步逐步推进，以便持续刷新画面。

## 主要参数说明

| 参数 | 含义 |
| --- | --- |
| `drone_count` | 无人机数量 |
| `target_count` | 目标数量 |
| `obstacle_count` | 障碍物数量 |
| `sensor_range` | 无人机基础感知半径 |
| `sensor_fov_deg` | 传感器视场角 |
| `comm_range` | 通信半径 |
| `packet_loss_rate` | 基础通信丢包率 |
| `comm_delay_steps` | 通信延迟步数 |
| `drone_speed` | 无人机基础速度 |
| `target_speed` | 目标基础速度 |
| `sensor_noise` | 传感器测量噪声 |
| `merge_radius` | 融合聚类半径 |
| `min_observations` | 目标确认所需最少观测数 |
| `fault_rate` | 无人机随机故障率 |
| `dispatch_ratio` | 派机比例上限的基础控制量 |
| `run_steps` | 仿真运行步数 |
| `seed` | 基础场景随机种子 |
| `execution_mode` | 运行模式，如三策略对比、仅 RADS、仅随机、仅全量 |
| `weather_preset` | 天气预设，如晴空、雾霾、降雨、风暴 |

需要注意的是，页面中的随机种子主要控制基础场景分布。每次点击“运行仿真”时，后端会额外注入运行时随机扰动，因此即使参数相同，目标轨迹、通信结果、故障过程和最终指标也可能不同。这种设计更接近真实系统中的不确定性。

## 指标解释

| 指标 | 含义 | 趋势 |
| --- | --- | --- |
| 严格成功率 | 同时满足有效确认、误差达标和融合一致性的任务比例 | 越高越好 |
| 定位达标率 | 融合误差低于阈值的任务比例 | 越高越好 |
| 确认率 | 有效观测数量达到任务需求的比例 | 越高越好 |
| 覆盖率 | 目标获得至少一次有效观测的比例 | 越高越好 |
| 平均融合误差 | 估计位置与真实位置的平均距离 | 越低越好 |
| 累计能耗 | 集群移动、感知、通信和空闲消耗总和 | 越低越好 |
| 平均计算耗时 | 每个仿真步的平均计算时间 | 越低越好 |
| 平均派机数 | 每步平均参与任务的无人机数量 | 需结合效果分析 |
| 平均信息时效 | 观测数据从采集到参与融合的平均延迟 | 越低越好 |

这些指标可以从不同角度解释算法价值：RADS 不一定追求派机数量最多，而是希望在有限资源下选择更合适的节点，使感知效果、能耗和信息时效达到更均衡的结果。

## 论文与答辩价值

本平台可以直接支撑毕业论文中的多个章节：

- 系统需求分析：展示平台面向多无人机协同感知任务，具备参数配置、仿真运行、结果展示和数据导出能力。
- 系统设计：说明前后端分层结构、会话式仿真流程、仿真内核和可视化模块。
- 算法设计：解释 RADS 动态子群调度的任务需求计算、效用函数、预算控制和贪心选择过程。
- 协同感知设计：说明多节点观测、噪声建模、通信丢包、延迟处理和加权融合机制。
- 实验分析：通过 RADS、随机派遣、全量派遣的对照结果分析成功率、误差、能耗和派机规模。
- 答辩演示：页面可以实时展示算法运行过程，比静态图表更容易说明系统完成度。

## 当前特点与局限

当前版本已经实现了较完整的毕设演示闭环，包括参数配置、复杂场景建模、三策略对比、3D 可视化、任务级表格和 CSV 导出。它适合作为毕业设计系统、论文实验支撑和答辩展示平台。

同时，它仍然属于仿真原型，不等同于真实无人机系统。当前模型中的无人机动力学、传感器模型、通信模型和障碍物地图都经过简化，主要目标是体现协同调度和态势感知算法思想。若后续继续扩展，可以考虑接入更真实的地图数据、飞行控制约束、通信网络仿真、深度学习目标检测模型或真实无人机接口。

## 快速总结

这个毕设平台的核心价值在于：把“无人机集群如何在复杂环境下动态组织子群完成多目标态势感知”这一问题，做成了一个可以运行、可以观察、可以调参、可以对比、可以导出实验数据的完整系统。RADS 动态子群策略是算法主线，Web 端可视化是展示主线，实验指标对比是论文论证主线。三者结合后，既能体现工程实现能力，也能支撑毕业论文中的方法解释和实验分析。
